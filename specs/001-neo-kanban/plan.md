# Implementation Plan: Neo-Kanban — AI-Agentic Project Management Platform

**Branch**: `003-core-kanban-flow` | **Date**: 2026-05-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-neo-kanban/spec.md`

---

## Summary

Neo-Kanban is a single-user AI-agentic project management platform where a Project Owner (PO)
guides an AI pipeline through Kanban columns — from natural-language Intent to reviewed, approved
code — with mandatory Human-in-the-Loop (HIL) checkpoints at every consequential decision gate.

Phase 1 delivers the core agentic loop: project management, spec/plan document generation via an
Architect Agent, a Kanban board (drag-to-trigger), and a Coder Agent that produces diffs reviewed
through Monaco's Diff Viewer.

Phase 2 adds real-time observability (WebSocket Thought Streaming), interrupt control (Pause &
Steer), persistent memory (MEMORY.md), structural codebase understanding (tree-sitter mapping),
and enriched Git/diff workflows (auto-branch, squash-merge, inline review comments).

**Project Type**: Web application — fullstack (React SPA + FastAPI JSON API)

**Constraints**: Single-user MVP; sandbox has no outbound network; each command timeout = 60 s; full task timeout = 10 min
**Scale/Scope**: 1 PO · multiple projects · ≤ 500 files per codebase (MVP)

## LangGraph State Machine

States flow strictly forward. Each `→ [HIL]` step calls `interrupt()` and persists state to DB
before yielding control. The graph resumes only when the API receives an explicit PO action.

```
┌──────────────────────────────────────────────────────────────────┐
│                      LangGraph State Machine                      │
│                                                                  │
│  IDLE ──[PO submits Intent]──► SPEC_GENERATION                   │
│                                       │                          │
│                              Agent generates SPEC.md             │
│                                       │                          │
│                              ◄── [HIL: PO reviews] ──►           │
│                            Approve           Request Revision     │
│                              │                     │             │
│                              ▼                     └──► SPEC_GENERATION
│                      PLAN_GENERATION                             │
│                              │                                   │
│                      Agent generates PLAN.md                     │
│                              │                                   │
│                     ◄── [HIL: PO reviews] ──►                   │
│                   Approve       Request Revision                  │
│                     │                 │                          │
│                     ▼                 └──► PLAN_GENERATION       │
│              TASK_BREAKDOWN                                      │
│                     │                                            │
│             Tasks written to DB → To Do                         │
│                     │                                            │
│                   IDLE ◄─────────────────────────────────────────┘
│
│  IDLE ──[PO drags task to In Progress]──► CODING
│                                               │
│                                   Agent reads context + codes    │
│                                               │
│                                     ◄── [HIL: PO reviews diff] ──►
│                                   Approve          Reject + feedback
│                                     │                    │
│                                     ▼                    └──► CODING
│                                 REVIEWING (optional Reviewer Agent)
│                                     │
│                                   DONE
└──────────────────────────────────────────────────────────────────┘
```

**HIL implementation**: Each checkpoint calls LangGraph `interrupt()`, which serialises the
current `AgentState` to PostgreSQL via LangGraph's checkpointer. The FastAPI endpoint for
`approve` / `revise` / `reject` updates the DB record and calls `graph.invoke()` with the
updated state to resume. No auto-continue is possible without an explicit API call.

## Database Schema — Phase 1

```sql
-- ─────────────────────────────────────────────
-- Core entities
-- ─────────────────────────────────────────────

CREATE TABLE projects (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(255) UNIQUE NOT NULL,
    description     TEXT,
    primary_language VARCHAR(50) NOT NULL,
    constitution    TEXT NOT NULL DEFAULT '',
    status          VARCHAR(20) NOT NULL DEFAULT 'active'
                        CHECK (status IN ('active', 'archived')),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE documents (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id  UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    type        VARCHAR(10) NOT NULL CHECK (type IN ('SPEC', 'PLAN')),
    content     TEXT NOT NULL DEFAULT '',
    status      VARCHAR(30) NOT NULL DEFAULT 'draft'
                    CHECK (status IN ('draft', 'approved', 'revision_requested')),
    version     INTEGER NOT NULL DEFAULT 1,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE tasks (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id  UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    title       VARCHAR(500) NOT NULL,
    description TEXT,
    status      VARCHAR(20) NOT NULL DEFAULT 'todo'
                    CHECK (status IN ('todo','in_progress','review','done','rejected','conflict')),
    priority    INTEGER NOT NULL DEFAULT 0,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    -- WIP constraint: at most 1 in_progress per project
    CONSTRAINT one_in_progress_per_project
        EXCLUDE USING btree (project_id WITH =)
        WHERE (status = 'in_progress')
);

CREATE TABLE agent_runs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id         UUID REFERENCES tasks(id) ON DELETE SET NULL,
    project_id      UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    agent_type      VARCHAR(20) NOT NULL
                        CHECK (agent_type IN ('architect','coder','reviewer')),
    agent_version   VARCHAR(20) NOT NULL DEFAULT '1.0.0',
    status          VARCHAR(20) NOT NULL DEFAULT 'running'
                        CHECK (status IN ('running','success','failure','awaiting_hil','paused')),
    input_artifacts TEXT[] NOT NULL DEFAULT '{}',
    output_artifacts TEXT[] NOT NULL DEFAULT '{}',
    started_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ,
    result          JSONB
);

CREATE TABLE diffs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id         UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
    original_content TEXT,
    modified_content TEXT,
    agent_run_id    UUID REFERENCES agent_runs(id) ON DELETE SET NULL,
    content         TEXT NOT NULL,
    files_affected  TEXT[] NOT NULL DEFAULT '{}',
    review_status   VARCHAR(10) NOT NULL DEFAULT 'pending'
                        CHECK (review_status IN ('pending','approved','rejected')),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE intents (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id  UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    content     TEXT NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_intents_project_id ON intents(project_id);

CREATE TABLE feedbacks (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id      UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    reference_type  VARCHAR(10) NOT NULL CHECK (reference_type IN ('document','task')),
    reference_id    UUID NOT NULL,
    content         TEXT NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ─────────────────────────────────────────────
-- Audit log — immutable, written BEFORE the action
-- ─────────────────────────────────────────────

CREATE TABLE audit_logs (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id            VARCHAR(100) NOT NULL,
    agent_version       VARCHAR(20) NOT NULL,
    action_type         VARCHAR(100) NOT NULL,
    action_description  TEXT NOT NULL,
    timestamp           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    input_refs          TEXT[] NOT NULL DEFAULT '{}',
    output_refs         TEXT[] NOT NULL DEFAULT '{}',
    result              VARCHAR(15) NOT NULL
                            CHECK (result IN ('success','failure','awaiting_hil')),
    project_id          UUID REFERENCES projects(id) ON DELETE SET NULL,
    task_id             UUID REFERENCES tasks(id) ON DELETE SET NULL
);

-- Prevent any UPDATE or DELETE on audit_logs at DB level
CREATE RULE audit_logs_no_update AS ON UPDATE TO audit_logs DO INSTEAD NOTHING;
CREATE RULE audit_logs_no_delete AS ON DELETE TO audit_logs DO INSTEAD NOTHING;
```

## WebSocket Event Protocol (Phase 2)

### Server → Client events

```jsonc
// Base wrapper (all server events share this envelope)
{
  "event_type": "THOUGHT | TOOL_CALL | TOOL_RESULT | ACTION | ERROR | STATUS_CHANGE",
  "sequence_number": 42,         // monotonically increasing per task
  "timestamp": "2026-05-11T10:30:00Z",
  "task_id": "<uuid>",
  "agent_run_id": "<uuid>",
  "content": { /* see per-type detail below */ }
}

// THOUGHT — internal reasoning step
{ "event_type": "THOUGHT", "content": { "reasoning": "..." } }

// TOOL_CALL — before executing a tool
{ "event_type": "TOOL_CALL",  "content": { "tool_name": "read_file", "arguments": { "path": "src/x.py" } } }

// TOOL_RESULT — after tool returns
{ "event_type": "TOOL_RESULT", "content": { "tool_name": "read_file", "result": "...", "success": true } }

// ACTION — side-effecting step (file write, git op, sandbox command)
{ "event_type": "ACTION", "content": { "action_type": "write_file", "description": "...", "target": "src/x.py" } }

// ERROR — recoverable or terminal error
{ "event_type": "ERROR", "content": { "error_type": "TimeoutError", "message": "...", "recoverable": false } }

// STATUS_CHANGE — LangGraph state transition
{ "event_type": "STATUS_CHANGE", "content": { "from": "CODING", "to": "REVIEWING", "reason": "..." } }
```

### Client → Server messages

```jsonc
// Pause request — agent finishes current step then halts
{ "type": "PAUSE", "timestamp": "2026-05-11T10:31:00Z" }

// Resume with optional steering
{ "type": "RESUME", "steering_instructions": "Focus on writing tests", "timestamp": "..." }

// Reconnect catch-up — server replays missed events from DB
{ "type": "CATCH_UP", "last_sequence": 41 }
```

## MEMORY.md Entry Format (Phase 2)

Each memory entry in `MEMORY.md` follows this structure. The file is generated from `memory_entries`
DB rows and is the artifact read by the Agent on each new task.

```markdown
## [MEMORY-{task_id_short}] {summary}

- **Timestamp**: 2026-05-11T10:30:00Z
- **Task ID**: {task_id}
- **Summary**: {one-line summary of what was implemented}
- **Files Affected**: `src/models/user.py`, `src/api/auth.py`
- **Lessons Learned**: {specific patterns, pitfalls, decisions — written for the next Agent to act on}
```

Required fields: `timestamp`, `task_id`, `summary`, `files_affected`, `lessons_learned`.
Entries are appended in chronological order. Blank lines separate entries. The Agent context
builder reads the full file and includes it verbatim in the system prompt.

## Technical Risks

### Phase 1

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| LangGraph `interrupt()` blocks async event loop | Medium | High | Run LangGraph graph in a separate thread (`asyncio.run_in_executor`); validate with an early spike |
| WIP race condition (two concurrent move requests) | Low | High | PostgreSQL `EXCLUDE` constraint is atomic; service-level check is defence-in-depth only |
| Sandbox escape (agent writes outside sandbox root) | Medium | Critical | `write_file` tool validates path starts with sandbox root; reject if not; add unit test |
| Monaco Diff format mismatch (git unified diff ≠ Monaco model) | Medium | Medium | Implement diff conversion utility early (Sprint 1); test with sample diffs before M7 |
| LangChain/LangGraph version incompatibility | Low | High | Pin exact versions in `requirements.txt`; run `pip-compile` to lock transitive deps |

### Phase 2

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Pause signal not async-safe across multiple Uvicorn workers | High | High | Store pause flag in Redis (`SET pause:{task_id} 1`); each node checks Redis at entry — process-independent |
| tree-sitter language bindings version mismatch | Medium | Medium | Pin `tree-sitter==0.23.0` and matching grammar packages; test on CI before P2-M4 |
| MEMORY.md concurrent write (agent + PO editing simultaneously) | Medium | Medium | PO edit uses `SELECT FOR UPDATE` on `memory_entries`; agent appends a new row (never updates existing) |
| WebSocket sequence gap on crash (missed events) | Low | High | Persist every event to `stream_events` table with `sequence_number`; `CATCH_UP` message replays from DB |
| Monaco inline comment line drift after squash/rebase | Medium | Low | Store line numbers from unified diff positions (not absolute file lines); diff positions are stable within the same diff object |
| tree-sitter parse error on malformed source files | Low | Low | Wrap parser in try/except; skip unparseable files and log warning; map continues with partial results |
